# مواصفات نظام المحاسبة والمخزون المتكامل  
**المشروع:** container-app — **الإصدار:** 1.1 (وثيقة عرض + فجوات + Backlog بناء)  
**التاريخ:** 2026-03-31

---

## 1) ملخص تنفيذي

يهدف هذا المستند إلى تعريف **نظام محاسبي عام (GL)** و**نظام مخزون** متكاملين، بما يتوافق لاحقاً مع **صفحة الحاويات** كمصدر تشغيلي للبيانات (فواتير، تكاليف، أصناف، أطراف).  
المرحلة الحالية: **تثبيت المواصفات والبنية التحتية للبيانات**؛ ربط الحاويات يتم وفق **نقاط ترحيل (Posting)** واضحة بعد اعتماد العميل على الخارطة الزمنية.

**مبادئ غير قابلة للتفاوض في التصميم**

- **قيد مزدوج**: مجموع المدين = مجموع الدائن لكل قيد (بالعملة الوظيفية بعد التحويل).
- **أثر مرجعي**: كل سطر قيد يرتبط بمصدر منطقي (يدوي، فاتورة، حاوية، مخزون، …) عند الترحيل.
- **فترات محاسبية**: لا ترحيل على فترة **مقفلة** أو **مؤمّنة**.
- **عدم حذف القيود المرحّلة**: التصحيح عبر **قيد عكسي** أو **إلغاء مرحّل** بصلاحية + سجل تدقيق (يُكمّل لاحقاً في الواجهة).

## 4) خطة الربط مع صفحة الحاويات (لاحقاً — بعد اعتماد المواصفات)

| حدث تشغيلي | أثر محاسبي مقترح | مصدر الترحيل |
|-------------|------------------|----------------|
| فاتورة شراء مرتبطة بحاوية | مخزون/مصروف/ذمم مورد حسب سياسة التقييم | `PURCHASE_VOUCHER` + `containerId` |
| فاتورة بيع مرتبطة بحاوية | إيراد/تكلفة بضاعة مباعة/ذمم عميل | `SALE_VOUCHER` + `containerId` |
| سطر تكلفة على الحاوية | مصروف مباشر أو إلى WIP حسب السياسة | `CONTAINER_COST` (امتداد) |
| إيراد/مصروف عام (`IncomeOutcomeEntry`) | قيد على حسابات نقدية/مصروف/إيراد | `INCOME_OUTCOME` |
| حركة مخزون ناتجة عن إغلاق شحنة | صرف/استلام حسب الإعداد | `INVENTORY` + مرجع `containerId` |

**قاعدة:** لا يُستبدل إدخال الحاوية بإدخال محاسبي مزدوج يدوي؛ يُفضّل **ترحيل واحد** من الوثيقة المعتمدة.

## 6) مخاطر يجب إبرامها مع العميل

1. **العملة الوظيفية** والسياسة متعددة العملات.  
2. **طريقة تقييم المخزون** (متوسط مرجّح مقابل FIFO — FIFO يحتاج طبقة دفعات).  
3. **الضريبة**: وجودها، نسبها، الاستثناءات.  
4. **سياسة تعديل الوثيقة** بعد الترحيل (ممنوع / قيد تعويضي / إلغاء ترحيل).  
5. **الترحيل التلقائي مقابل اليدوي** لكل نوع وثيقة.

## 7) الملحق التقني — نماذج البيانات المضافة في المشروع

تمت إضافة جداول أولية في `schema.prisma` (دون إزالة النماذج الحالية):

- `GlAccount` — دليل الحسابات  
- `FiscalYear`, `FiscalPeriod` — السنة والفترات  
- `JournalEntry`, `JournalLine` — القيود والأسطر (مع ارتباطات اختيارية: `Party`, `Container`, `Store`)  
- `InvWarehouse`, `InvStockMove`, `InvStockBalance` — مخزون وحركات وأرصدة  

**النماذج القديمة** `AccountingMove` / `AccountingLine` تبقى كما هي إلى حين ترحيل بيانات أو إيقافها بقرار صريح.

**الخطوة التالية للمطور:** تطبيق الهجرة على قاعدة التطوير (تم إنشاء ملف الهجرة `prisma/migrations/…_add_gl_and_inventory/`):

```bash
cd backend
npx prisma migrate dev
npx prisma generate
```

## 8) ما تُعرض على العميل في الاجتماع

1. جدول الوحدات (أقسام 2 و 3).  
2. خطة الربط مع الحاويات (قسم 4).

## 9) تحليل الفجوات — ما كان ناقصاً في الخطة السابقة (مُكمَّل الآن)

هذا القسم يجمع **مخرجات بحث/مراجعة** مقارنةً بأنظمة محاسبة ومخازن ناضجة ومع **النموذج الحالي في المشروع** (`schema.prisma`). الهدف: لا يُفاجَأ الفريق لاحقاً بمتطلبات «مفترضة». كل بند إما يُدرَج في مرحلة بناء أو يُستبعد صراحة مع العميل.

### 9.1 محاسبة — فجوات وظيفية

| البند | الوصف | حالة المشروع حالياً | مقترح المرحلة |
|--------|--------|----------------------|----------------|
| **كيان قانوني / شركة** | أكثر من منشأة، فروع، تقارير منفصلة أو مدمجة | غير مُمثَّل | **P3 أو حسب الحاجة**: جدول `LegalEntity`/شركة + ربط مستقبلي للقيود |
| **جدول عملات رسمي** | رموز ISO، عدد الخانات العشرية، العملة الوظيفية للكيان | جزئي: `ExchangeRateSnapshot` بدون جدول عملات ولا ربط يومي إلزامي | **P0–P1**: `Currency` + `FxRate` (تاريخ، زوج، مصدر) |
| **ترقيم وثائق** | سلاسل أرقام لكل سنة/نوع (قيد، سند، فاتورة) بدون تضارب | غير مُنمَّذج | **P0**: جدول `DocumentSequence` أو توليد ذري في الخدمة |
| **ذمم تفصيلية (Open Items)** | كل فاتورة/دفعة كعنصر مستقل للمطابقة وليس مجرد رصيد حساب | غير موجود؛ `Party` + قيود فقط | **P1–P2**: `ArOpenItem` / `ApOpenItem` أو ربط قوي بـ `SaleVoucher`/`Purchase…` |
| **سند قبض / صرف كوثيقة** | ربط الدفعة بعدة فواتير، تخصيم، فروقات صرف | غير موجود كنموذج مستقل | **P2**: `Payment`/`Receipt` + تطبيق على الفواتير |
| **حسابات بنكية وسياسات** | IBAN، عملة الحساب، تسويات | غير موجود | **P2–P3**: `BankAccount` + حركة بنك |
| **تسوية بنكية** | استيراد كشف، مطابقة تلقائية/يدوية، عناصر معلّقة | غير موجود | **P3+**: `BankStatement` + `BankStatementLine` + مطابقة |
| **قيود متكررة** | إيجار شهري، إهلاك… | غير مذكور | **P3**: `RecurringJournalTemplate` |
| **توزيع تكاليف (Allocations)** | توزيع مصروف على أبعاد/مشاريع | غير موجود | **P3** حسب الطلب |
| **إقرارات ضريبية مفصّلة** | قنوات إخراج، ملفات سلطة ضريبية | مذكور كـ«حسب التشريع» فقط | **P4** + مواصفة بلد العميل |
| **موافقات وسير عمل** | من يرسل للترحيل، من يعتمد | غير موجود | **P1–P2**: `ApprovalStep` أو تكامل بسيط بـ `JournalEntry.status` |
| **سجل تدقيق شامل** | من غيّر ماذا ومتى (قبل/بعد) | جزئي: حقول `createdAt` فقط | **P0–P1**: جدول `AuditLog` (كيان، معرّف، حقل، قديم، جديد، مستخدم) |
| **فصل مهام (SoD)** | من يُنشئ القيد ≠ من يُرحّل | `UserRole` بسيط | **P1**: أدوار محاسبة مفصّلة |
| **ترحيل/إلغاء ترحيل آمن تحت التزامن** | قفل تفاؤلي عند الترحيل لمنع ازدواج الترقيم | غير مطبّق | **P0** تصميم خدمة ترحيل + معاملات DB |

### 9.2 مخزون — فجوات وظيفية

| البند | الوصف | حالة المشروع حالياً | مقترح المرحلة |
|--------|--------|----------------------|----------------|
| **وحدات قياس وتحويل** | علبة، كرتون، كغم بمعاملات | `Item` بدون UoM | **P2**: `UnitOfMeasure` + `ItemUomConversion` |
| **سياسة السالب** | السماح أو منع الرصيد السالب | غير مُعرَّفة | **P2** قاعدة على مستوى الصنف/المخزن |
| **FIFO / دفعات (LOT)** | تتبع تكلفة حسب الدفعة | مذكور W8؛ غير مُنمَّذج | **P3** بعد ثبوت WAC في P2 |
| **تكلفة مُتَرَاكَمة (Landed cost)** | شحن/جمارك تُوزّع على تكلفة الشراء | منطق الحاوية قريب لكن غير موحّد مع المخزن | **P3** ربط تكاليف حاوية → استلام |
| **حجز مخزون** | حجز لبيع معلّق | غير موجود | **P3** حسب حاجة المبيعات |
| **تجميع BOM** | تفكيك/تجميع أصناف | غير في النطاق الحالي | **خارج النطاق أو P4** |
| **وثيقة جرد كاملة** | رأس + أسطر + حالات | غير مُنمَّذجة | **P2**: `StockCount` + أسطر + ترحيل فرق |
| **تكامل عكسي مع أسطر الحاوية** | متى يُنشَأ `Item`/`حركة` من `ContainerLineItem` | غير مُحدَّد | **P1 تصميم** ثم **P2 تنفيذ** |

### 9.3 تشغيل، تقنية، وجودة

| البند | الوصف | مقترح |
|--------|--------|--------|
| **نسخ احتياطي واستعادة** | سياسة DB، اختبار استعادة | تشغيل بنية البنية التحتية — وثيقة تشغيل |
| **استيراد افتتاحي** | Excel/CSV لأرصدة سنة أولى | أداة استيراد **P1** |
| **تصدير للمحاسب الخارجي** | CSV قيود، تنسيق تربية محلية | **P2** حسب الطلب |
| **اختبارات تلقائية** | ترحيل، توازن قيد، تقييم مخزون | **P0+** PHPUnit/Jest أو اختبارات تكامل |
| **UAT** | سيناريوهات موقعة بالعميل | نهاية كل مرحلة |

### 9.4 فجوات ربط «الحاويات ↔ المحاسبة» (مُفصَّلة)

| الفجوة | التوضيح | الإجراء |
|--------|---------|---------|
| **لا `containerId` على الفواتير في القيد** | القيد يدعم البُعد في `JournalLine` لكن الترحيل التلقائي غير مُنفَّذ | **P1**: خدمة `postSaleVoucher`، `postPurchaseVoucher` |
| **سطور تكلفة الحاوية** | `ContainerCostLine` بدون `JournalSourceType.CONTAINER_COST` في الـ enum بعد | **P1**: إضافة مصدر + سياسة قيد |
| **ازدواجية مع `IncomeOutcomeEntry`** | قد يُرحّل مرتين إن لم يُحدَّد «حالة ترحيل» على الوثيقة | **P1**: حقل `postedJournalEntryId` أو `postingStatus` على الجداول التشغيلية |
| **ازدواجية مع `AccountingMove` القديم** | مستخدمو قديم يتوقعون شاشة واحدة | قرار: إيقاف تدريجي أو مهاجرة بيانات **بعد P0** |

## 10) خطة البناء المحدّثة — Backlog مرتب حسب أولوية البدء

استخدم هذا القسم كـ **قائمة مهام** للسبرينتات؛ كل صف يترجم إلى تذاكر (DB + API + UI + اختبار).

### المرحلة P0 — أساس محاسبة صالح للإنتاج (يبدأ هنا)

1. تطبيق هجرة `Gl*` / `Journal*` / `Fiscal*` على البيئة.
2. **خدمة ترحيل** قيد: تحقق توازن `sum(debitBase) = sum(creditBase)`، فترة مفتوحة، حسابات ترحيل فقط.
3. **API + UI**: دليل حسابات (CRUD مقيّد)، سنة/فترات، قيد يدوي (مسودة/ترحيل).
4. تقارير: **ميزان مراجعة** و**دفتر أستاذ حساب** (لنطاق تاريخ).
5. **AuditLog** أساسي على `JournalEntry` / `JournalLine` (إنشاء/تعديل/ترحيل).
6. **DocumentSequence** لأرقام القيود.

### المرحلة P1 — ربط التشغيل الحالي

7. حقول `postingStatus` / `postedJournalEntryId` على `SaleVoucher`, `PurchaseInvoiceVoucher`, `IncomeOutcomeEntry` (حسب القرار).
8. قوالب ترحيل: شراء/بيع/إيراد-مصروف؛ ربط `sourceId` + `containerId` في `JournalLine`.
9. تقارير ذمم مبسطة من الفواتير الحالية (قبل نموذج Open Items الكامل إن لزم).
10. استيراد أرصدة افتتاحية (CSV) اختياري.

### المرحلة P2 — مخزون محاسبي

11. إدارة مخازن + سياسة سالب + UoM.
12. حركات واردة/صادرة + تحديث `InvStockBalance` (WAC).
13. وثيقة جرد + فرق + قيد تسوية.
14. ربط حركة شراء/بيع بالفواتير الحالية.

### المرحلة P3 — نقد، بنوك، تسوية مبدئية
